# OpenSpec 中文工作流指南

## 🎯 什么是OpenSpec？

OpenSpec是一个**结构化变更管理框架**，它将软件开发从"想到哪做到哪"的模式，转变为"思考→设计→实施→归档"的规范化流程。

### 核心理念

```
想法 💡 → 探索 🔍 → 提案 📋 → 规格 📐 → 设计 🏗️ → 任务 ✅ → 实施 🛠️ → 归档 📚
```

## 🔄 OpenSpec 工作流详解

### 阶段1: 探索思考 (Explore Mode)

**目的**: 在没有确定方向时深入思考问题
**命令**: `/opsx-explore`

```bash
# 使用场景
/opsx-explore  # 进入探索模式，思考但不保存

# 探索时可以思考的问题：
# - 这个功能真的需要吗？
# - 有哪些实现方式？
# - 会影响哪些现有功能？
# - 技术风险是什么？
```

**特点**:
- 🔍 **自由思考**: 不受变更框架约束
- 🧩 **问题分析**: 深入分析需求和困难点
- 💡 **创意发散**: 探索多种可能性
- 🗑️ **临时性质**: 不会永久保存，仅用于思考

### 阶段2: 创建变更 (Create Change)

**目的**: 为一个想法建立规范化容器
**命令**: `/opsx-new <变更名>`

```
变更目录结构：
openspec/changes/<变更名>/
├── .openspec.yaml      # 变更元信息
├── proposal.md         # 变更提案 (为什么做)
├── design.md          # 技术设计 (怎么做)
├── specs/             # 详细规格 (做什么)
│   └── <功能名>/
│       └── spec.md
└── tasks.md           # 实施任务清单 (具体步骤)
```

**两种创建方式**:

#### 方式1: 逐步创建
```bash
/opsx-new 实时价格监控  # 一步步创建每个工件
```

#### 方式2: 一键生成 (推荐新手)
```bash
/opsx-propose 实时价格监控  # 一次性生成所有工件
# 或
/opsx-ff 实时价格监控  # 快速生成所有工件
```

### 阶段3: 完善工件

#### 📋 变更提案 (proposal.md)
**回答**: "为什么要做这个变更？"

```markdown
## 为什么
[说明业务价值和用户需求]

## 做什么
[列出主要变更内容]

## 新增能力
[描述新功能]

## 修改能力
[描述现有功能的变更]

## 影响范围
[列出受影响的文件和模块]
```

#### 📐 详细规格 (specs/)
**回答**: "具体要做到什么程度？"

```markdown
## 新增需求

### 需求: <功能名称>

<功能描述>

#### 场景: <使用场景>
- **当** <触发条件>
- **那么** <期望结果>
- **并且** <附加条件>
```

#### 🏗️ 技术设计 (design.md)
**回答**: "要如何实现？"

```markdown
## 背景
[当前系统状态]

## 目标/非目标
**目标**:
- [要实现的功能]

**非目标**:
- [本次不做的事情]

## 技术决策
### 决策1: <架构选择>
[决策说明和理由]
```

#### ✅ 实施任务 (tasks.md)
**回答**: "具体要做什么？"

```markdown
## 1. 数据库层
- [ ] 1.1 创建价格监控表
- [ ] 1.2 添加索引优化查询

## 2. 后端服务
- [ ] 2.1 实现监控引擎
- [ ] 2.2 添加REST API

## 3. 验证
- [ ] 3.1 单元测试
- [ ] 3.2 集成测试
```

### 阶段4: 执行实施 (Apply)

**目的**: 按照任务清单实施变更
**命令**: `/opsx-apply <变更名>`

```bash
/opsx-apply 实时价格监控  # 开始实施
```

**实施过程**:
1. 📋 **查看任务**: 系统显示所有待办任务
2. 🛠️ **逐一完成**: 按顺序完成每个任务
3. ✅ **标记完成**: 完成后自动标记
4. 🔄 **持续跟进**: 可随时暂停和继续

### 阶段5: 验证检查 (Verify)

**目的**: 确保实现符合规格要求
**命令**: `/opsx-verify <变更名>`

```bash
/opsx-verify 实时价格监控  # 验证实现
```

**检查内容**:
- 📊 **功能完整性**: 是否实现了所有规格要求
- 🏗️ **技术一致性**: 是否遵循设计方案
- ✅ **任务完成度**: 所有任务是否正确完成
- 🔍 **质量检查**: 代码质量和测试覆盖

### 阶段6: 归档保存 (Archive)

**目的**: 将完成的变更永久保存
**命令**: `/opsx-archive <变更名>`

```bash
/opsx-archive 实时价格监控  # 归档变更
```

**归档后**:
- 📁 **移动位置**: 变更从 `changes/` 移到 `changes/archive/`
- 📅 **日期标记**: 自动添加日期前缀
- 📚 **历史记录**: 成为项目决策历史的一部分
- 🔍 **随时查阅**: 可随时查看历史决策

## 🎯 在您的工作中的应用

### 根据您的项目特点:

#### 📊 多组件架构 (C++ + Node.js)
```bash
# 涉及多个模块的复杂变更
/opsx-explore  # 先思考接口设计
/opsx-new 跨模块功能  # 创建变更
# 重点：详细设计模块间接口
```

#### ⚡ 性能敏感系统
```bash
# 性能优化类变更
/opsx-new 性能优化
# 重点：基准测试目标和性能指标
```

#### 🌐 API限流控制
```bash
# 涉及外部API的变更
/opsx-new 新数据源集成
# 重点：限流策略和错误处理
```

### 常见场景对应的工作流:

#### 🏗️ 新功能开发
```bash
/opsx-explore  # 功能可行性分析
/opsx-propose 新功能名  # 详细提案
/opsx-apply 新功能名  # 实施开发
/opsx-archive 新功能名  # 完成归档
```

#### 🐛 问题修复
```bash
/opsx-explore  # 问题根因分析
/opsx-new 修复问题名  # 快速创建
/opsx-apply 修复问题名  # 实施修复
```

#### 🔄 重构优化
```bash
/opsx-explore  # 重构范围评估
/opsx-new 重构模块名  # 分阶段计划
/opsx-apply 重构模块名  # 逐步实施
```

## 👥 与我的协作方式

### 我的角色定位:
- 🎯 **思考伙伴**: 帮助您分析问题和需求
- 📝 **文档助手**: 生成规范的变更工件
- 🛠️ **实施引擎**: 按照计划执行具体任务
- 🔍 **质量检查**: 确保实现符合设计要求

### 您的角色:
- 💡 **需求定义**: 明确要做什么，为什么做
- 🎯 **决策确认**: 在关键节点做出技术决策
- ✅ **质量把关**: 审查生成的工件和代码
- 📚 **知识沉淀**: 维护项目的决策历史

### 协作流程示例:

```bash
您: 我想添加实时价格监控功能

我: 好的！让我们先探索一下这个需求...
    (进入 /opsx-explore 模式)

我: 基于探索结果，我创建以下变更提案：
    - 变更名: 实时价格监控
    - 主要功能: 持续监控价格变动
    - 技术方案: 多线程监控 + 通知推送
    
您: 看起来不错，继续实施

我: 好的！开始实施...
    (进入 /opsx-apply 模式)
    
我: ✅ 任务1: 数据库表创建 - 完成
    ✅ 任务2: 监控引擎实现 - 完成
    🔄 任务3: 通知接口开发 - 进行中...

我: 所有任务完成！验证一下...
    (/opsx-verify)

我: 变更实施完整，准备归档...
    (/opsx-archive)
```
