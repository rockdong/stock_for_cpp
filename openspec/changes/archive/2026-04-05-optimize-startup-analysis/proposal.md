# 启动分析检查优化提案

## Why

每次程序重启时，系统都会强制重新执行完整的股票分析流程，即使当天数据已经完整分析完毕。这导致：
- **资源浪费**：重复调用 Tushare API，消耗限流配额（500次/分钟限制）
- **时间浪费**：分析数千只股票需要数小时，重启后再次等待相同时间
- **运维效率低**：频繁重启（如配置更新、故障恢复）触发不必要的重复计算
- **API成本**：Tushare Pro API 有积分限制，重复调用影响后续数据获取

**为什么现在解决**：这个问题在每次重启时都会触发，严重影响日常运维效率和系统稳定性。已有分析进度表和数据记录，但启动流程未利用这些信息进行智能判断。

## What Changes

### 主要变更
- 在程序启动时增加智能检查机制，判断当天是否已完成完整分析
- 实现双重验证：文件记录检查（快速） + 数据库进度表检查（准确）
- 启动流程优化：
  - **场景1**：当天已完成分析 → 跳过首次执行，直接进入调度等待
  - **场景2**：当天未完成分析 → 立即执行一次完整分析
  - **场景3**：`--once` 模式 → 强制执行，不受检查限制
- 增强日志记录：明确记录跳过/执行的原因和依据

### 兼容性保障
- 不改变现有分析逻辑和数据结构
- 不影响定时调度器的核心功能
- 保持 `--once` 强制执行参数的能力
- 与现有交易日判断和跨天逻辑兼容

## Capabilities

### New Capabilities

- **`startup-analysis-check`**: 启动分析检查规范
  - 定义启动前的检查流程和判断逻辑
  - 规定文件记录和数据库进度的双重验证机制
  - 明确各种启动场景的处理策略

### Modified Capabilities

- **`analysis-process-storage`**: 分析过程存储规范（扩展）
  - 新增：利用 `analysis_progress` 表状态进行完整性验证
  - 新增：检查进度表 `status` 字段和计数器一致性
  - 保持：现有存储逻辑和 JSON 结构不变

## Impact

### 受影响的代码文件
- **`cpp/main.cpp`**（主要修改）
  - 第 619-621 行：启动流程增加智能检查
  - 新增辅助函数：检查文件记录和数据库进度状态
  
- **`cpp/scheduler/Scheduler.h/cpp`**（扩展）
  - 新增公共方法：`hasRunToday()` 暴露给外部调用
  - 保持：现有调度逻辑不变

- **`cpp/data/database/AnalysisProcessRecordDAO.h/cpp`**（扩展）
  - 新增方法：查询最新分析进度状态
  - 保持：现有 upsert 和查询逻辑不变

### 不受影响的系统
- API 接口层（Node.js）：无需修改
- 前端界面：无需修改
- 数据库表结构：无需修改
- 定时调度核心逻辑：无需修改

### 系统依赖关系
- 依赖现有：`data/.last_run` 文件记录机制
- 依赖现有：`analysis_progress` 数据库表
- 依赖现有：`calculateAnalysisDate()` 跨天判断函数
- 依赖现有：`TradeCalendar::isTradingDayToday()` 交易日判断

### 验证场景
1. **当天已完成 → 重启**：跳过执行，进入调度等待
2. **当天未完成 → 重启**：立即执行完整分析
3. **执行中断 → 重启**：检测到部分完成，重新执行
4. **跨天重启（凌晨前后）**：正确判断分析日期
5. **非交易日重启**：跳过执行（已有交易日判断）
6. **`--once` 模式**：强制执行，不受检查限制
