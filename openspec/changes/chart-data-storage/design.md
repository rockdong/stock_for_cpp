## Context

当前系统在执行股票分析时，会计算 EMA17、EMA25 等技术指标，但这些指标值只在内存中临时使用，分析完成后即丢弃。用户只能看到最终的策略判断结果（买入/卖出/持有），无法查看分析时的具体指标位置和价格走势，难以验证策略判断的准确性。

系统架构：
- C++ 引擎：负责数据获取、指标计算、策略分析
- Node.js 服务：负责飞书消息处理和用户交互
- SQLite 数据库：共享数据存储

## Goals / Non-Goals

**Goals:**
- 存储 EMA17、EMA25 指标值和 OHLC 价格数据
- 支持 K 线图生成，叠加 EMA 指标线
- 用户可通过飞书命令查询股票图表
- 支持日线、周线、月线三种频率

**Non-Goals:**
- 不存储全部历史数据，仅保留最近 10 条
- 不支持自定义指标参数（固定 EMA17/EMA25）
- 不实现图表缓存机制（每次查询实时生成）

## Decisions

### 决策 1: 数据存储方式 - JSON 字段存储

**选择**: 使用 JSON 字段存储 10 条 K 线数据

**理由**:
- 查询简单：一次查询获取所有数据，无需 JOIN
- 结构灵活：可存储多个指标值
- 数据量可控：每只股票每频率每天仅 1 条记录

**备选方案**:
- 方案 A: 每条记录一天数据（关系型）→ 查询需要聚合，复杂度高
- 方案 B: JSON 存储（当前选择）→ 简单直接

### 决策 2: 图表生成位置 - Node.js 端

**选择**: 在 Node.js 端使用 lightweight-charts 生成图表

**理由**:
- Node.js 图表库生态丰富
- C++ 图表库较少，集成复杂
- 图表生成与飞书发送在同一进程，避免跨进程通信

**备选方案**:
- 方案 A: C++ 生成图片 → 库少，依赖重
- 方案 B: Node.js 生成（当前选择）→ 生态好，维护简单

### 决策 3: 图表库选择 - lightweight-charts

**选择**: TradingView 开源的 lightweight-charts

**理由**:
- 专为金融图表设计，K 线图原生支持
- 支持 Node.js 环境渲染
- 轻量级，性能优秀

**备选方案**:
- Chart.js: K 线图需要插件，不够专业
- Plotly.js: 功能强大但体积大
- lightweight-charts（当前选择）: 专业金融图表库

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| JSON 字段查询性能 | 保持数据量小（10 条），添加索引 |
| lightweight-charts Node.js 支持有限 | 使用 node-canvas 作为渲染后端 |
| 图片生成内存占用 | 限制并发图表生成数量 |
| 数据库记录增长 | 定期清理超过 30 天的历史记录 |

## Migration Plan

**部署步骤**:
1. C++ 编译部署（新增 ChartDataDAO）
2. 数据库自动创建 `chart_data` 表
3. Node.js 部署（新增图表生成模块）
4. 飞书命令生效

**回滚策略**:
- 删除 `chart_data` 表
- 移除 Node.js 图表相关代码
- 无数据迁移风险
