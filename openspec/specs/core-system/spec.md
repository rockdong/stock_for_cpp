# 核心系统规格说明

## 系统需求

### 功能性需求

#### 股票数据管理
- **需求**: 系统必须能够获取、存储和查询股票基础信息
  - **场景**: 获取股票列表
    - **当** 用户请求股票列表时
    - **那么** 系统返回包含股票代码、名称、行业等完整信息
    - **并且** 响应时间不超过2秒
  
  - **场景**: 查询特定股票信息
    - **当** 用户提供股票代码时
    - **那么** 系统返回该股票的详细基础信息
    - **并且** 支持模糊搜索和代码格式自动补全

#### 技术指标计算
- **需求**: 系统必须提供常用技术指标的计算功能
  - **场景**: 计算MACD指标
    - **当** 用户请求某股票的MACD数据时
    - **那么** 系统返回DIF、DEA、MACD柱状线等完整数据
    - **并且** 支持自定义时间周期和参数设置

  - **场景**: 计算移动平均线
    - **当** 用户请求均线数据时
    - **那么** 系统返回5日、10日、20日、60日等常用均线
    - **并且** 计算结果精度到小数点后2位

#### 策略回测引擎
- **需求**: 系统必须能够执行量化交易策略的回测
  - **场景**: 策略回测
    - **当** 用户定义交易策略并设定回测周期时
    - **那么** 系统执行完整回测并返回收益曲线、最大回撤等指标
    - **并且** 模拟交易成本和滑点等现实因素

#### 策略趋势判断窗口化
- **需求**: 系统必须支持可配置的趋势判断窗口参数
  - **场景**: 默认窗口判断
    - **当** 用户未指定趋势判断窗口参数
    - **那么** 系统使用默认值 trend_days=3
    - **并且** 只检查最近3天的 EMA 趋势
  
  - **场景**: 自定义窗口判断
    - **当** 用户指定趋势判断窗口参数 trend_days=5
    - **那么** 系统检查最近5天的 EMA 趋势
    - **并且** 所有指定天数内 EMA 必须呈上升趋势
  
  - **场景**: 窗口参数验证
    - **当** 用户设置 trend_days <= 0 或 trend_days > 数据长度
    - **那么** 系统返回 std::nullopt 表示无法判断
    - **并且** 记录警告日志说明参数无效

#### 趋势判断逻辑改进
- **需求**: 系统必须使用窗口化趋势判断替代全序列检查
  - **场景**: 窗口化趋势向上判断
    - **当** 最近 N 天（N=trend_days）EMA 值连续上升
      - 每天的 EMA 值严格大于前一天
    - **那么** 系统判定趋势向上，返回 true
  
  - **场景**: 窗口内趋势中断判断
    - **当** 最近 N 天内任意一天的 EMA 值 <= 前一天
    - **那么** 系统判定趋势不满足，返回 false
    - **并且** 不继续检查剩余天数
  
  - **场景**: 数据不足判断
    - **当** EMA 数据长度 < trend_days + 1
    - **那么** 系统返回 false
    - **并且** 记录调试日志说明数据不足

#### 策略信号输出结构
- **需求**: 系统必须为策略分析结果提供完整的信号信息
  - **场景**: 完整信号输出
    - **当** 策略生成买入或卖出信号
    - **那么** 系统返回 AnalysisResult 结构，包含：
      - ts_code: 股票代码
      - strategy_name: 策略名称
      - trade_date: 交易日期
      - label: 信号标签（买入/卖出/持有）
      - opt: 操作类型（buy/sell/hold）
      - freq: 频率（d/w/m）
      - strength: 信号强度（STRONG/MEDIUM/WEAK）
      - confidence: 置信度（0-100）
      - risk_warning: 风险提示文本
  
  - **场景**: 默认值兼容性
    - **当** 策略未提供信号强度和置信度
    - **那么** 系统使用默认值
      - strength = MEDIUM
      - confidence = 50.0
      - risk_warning = ""（空字符串）
    - **并且** 现有调用代码无需修改即可编译通过

### 非功能性需求

#### 性能需求
- **需求**: 系统必须满足高性能数据处理要求
  - **场景**: 大批量数据处理
    - **当** 系统处理1000只股票的技术指标计算时
    - **那么** 总处理时间不超过30秒
    - **并且** 内存使用不超过2GB

  - **场景**: 并发用户访问
    - **当** 50个用户同时访问系统时
    - **那么** API响应时间95%分位数不超过5秒
    - **并且** 系统CPU使用率不超过80%

#### 可靠性需求
- **需求**: 系统必须具备高可用性和容错能力
  - **场景**: API服务商故障
    - **当** Tushare API服务不可用时
    - **那么** 系统自动切换到备用数据源或缓存数据
    - **并且** 记录详细的故障日志和恢复时间

  - **场景**: 数据库连接异常
    - **当** SQLite数据库连接失败时
    - **那么** 系统优雅降级，提供只读缓存数据
    - **并且** 自动重试连接机制最多尝试3次

### 安全性需求

#### 数据安全
- **需求**: 系统必须保证敏感数据的安全性
  - **场景**: API密钥保护
    - **当** 系统存储和使用第三方API密钥时
    - **那么** 密钥必须使用AES-256加密存储
    - **并且** 运行时解密过程中密钥不会明文暴露在内存中

  - **场景**: 用户数据隐私
    - **当** 处理用户查询和配置信息时
    - **那么** 敏感信息在日志中自动脱敏
    - **并且** 遵循数据最小化原则存储必要信息

#### 访问控制
- **需求**: 系统必须实现合理的访问控制机制
  - **场景**: API访问限制
    - **当** 外部用户访问API接口时
    - **那么** 系统实现基于IP和频率的访问控制
    - **并且** 记录所有访问日志用于安全审计

### 兼容性需求

#### 平台兼容性
- **需求**: 系统必须支持主流操作系统和容器环境
  - **场景**: 跨平台部署
    - **当** 在Ubuntu 20.04+和macOS 12+系统上部署时
    - **那么** 系统必须正常编译和运行
    - **并且** 所有功能模块保持一致的行为

  - **场景**: 容器化环境
    - **当** 在Docker容器中运行时
    - **那么** 系统支持linux/amd64和linux/arm64架构
    - **并且** 镜像大小不超过500MB

#### 数据兼容性
- **需求**: 系统必须支持数据格式的向后兼容
  - **场景**: 数据库结构升级
    - **当** 需要修改SQLite数据库表结构时
    - **那么** 系统提供自动迁移脚本
    - **并且** 保证现有数据不丢失的情况下完成升级
